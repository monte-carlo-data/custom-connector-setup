# Agent Instructions

## Objective

Implement every method in `integrations/<name>/integration.py` for your target database. Each method either returns a Jinja template string or implements a connection lifecycle hook. The file contains five classes with stub methods — your job is to replace every stub with a working implementation.

**A human implements `credential_env_vars` and `create_connection` (credential-sensitive methods). Agents implement everything else, starting with the remaining connection methods then Phase 2.**

## Rules

- **Only edit `integrations/<name>/integration.py`.** Do not modify any files in `tests/`, `conftest.py`, or the plugin. The tests are the spec — make your code pass them, not the other way around.
- **Add your database driver to `integrations/<name>/requirements.txt`** (e.g. `psycopg2-binary`, `snowflake-connector-python`) and install it before implementing `create_connection()`.
- Every template method must return a **Jinja template string**, not raw SQL and not Python logic. For example: `return "AVG({{ field }})"`.
- **Do not edit `capabilities.json`.** This file is auto-generated by the test suite after each run in `output/<name>/capabilities.json` — do not modify it manually.
- Read each method's docstring before implementing it. The docstring documents the Jinja variables the template receives, example implementations for common databases (Snowflake, PostgreSQL, BigQuery), and which metrics the method enables.
- **Never read `.env` files or files containing credentials.** The `.env` files in `integrations/<name>/` contain sensitive database credentials managed by humans. Agents must not read, display, or reference their contents.

## Prerequisites (Human Setup)

Before an agent begins, a human must:

1. Create `integrations/<name>/.env` with database credentials.
2. Add the database driver to `integrations/<name>/requirements.txt` and install it.
3. Implement `credential_env_vars` and `create_connection` in `BaseIntegration` — these two methods handle credentials and are human-only.

See [`prompts/implement-connection.md`](prompts/implement-connection.md) for details. The remaining 4 connection methods (`create_cursor`, `execute_query`, `fetch_all_results`, `close_connection`) can be implemented by an agent.

**Agents: after the human completes the prerequisites, implement the remaining connection methods, then verify the gate passes (`pytest -m connection` → 3 tests pass) before starting Phase 2.**

## Implementation

Follow [`prompts/implement-all.md`](prompts/implement-all.md) — it orchestrates all phases in order with gate checks between each:

1. **Phase 1: Connection** — [`prompts/implement-connection.md`](prompts/implement-connection.md) (human does credentials, agent does the rest)
2. **Phase 2: Metadata** — [`prompts/implement-metadata.md`](prompts/implement-metadata.md) (5 methods)
3. **Phase 3: Custom Monitors** — [`prompts/implement-monitors.md`](prompts/implement-monitors.md) (3 methods)
4. **Phase 4: Query Language** — [`prompts/implement-ql.md`](prompts/implement-ql.md) (~94 methods)

When tests fail, consult [`prompts/test-and-fix.md`](prompts/test-and-fix.md).
