# Agent Instructions

## Objective

Implement every method in `integrations/<name>/integration.py` for your target database. Each method either returns a Jinja template string or implements a connection lifecycle hook. The file contains five classes with stub methods â€” your job is to replace every stub with a working implementation.

## Credentials

Before you start, a human will provide an `integrations/<name>/.env` file containing the database credentials your integration needs. You do not need to create or guess credentials â€” they are already set.

Your first task is implementing `credential_env_vars()` in `BaseIntegration`. This method maps logical keys (e.g. `"host"`, `"password"`) to the environment variable names defined in `.env`. Check the integration's `.env` file for the variable names the human has configured.

Once `credential_env_vars()` and the other `BaseIntegration` connection methods are implemented, validate the setup by running:

```bash
INTEGRATION=<name> pytest -m connection
```

This runs two targeted tests that verify credentials are loaded and the connection is established. **Do not move on to templates until all connection tests pass.**

## Rules

- **Only edit `integrations/<name>/integration.py`.** Do not modify any files in `tests/`, `conftest.py`, or the plugin. The tests are the spec â€” make your code pass them, not the other way around.
- **Add your database driver to `integrations/<name>/requirements.txt`** (e.g. `psycopg2-binary`, `snowflake-connector-python`) and install it before implementing `create_connection()`.
- Every template method must return a **Jinja template string**, not raw SQL and not Python logic. For example: `return "AVG({{ field }})"`.
- **Do not edit `capabilities.json`.** This file is auto-generated by the test suite after each run in `output/<name>/capabilities.json` â€” do not modify it manually.
- Read each method's docstring before implementing it. The docstring documents the Jinja variables the template receives, example implementations for common databases (Snowflake, PostgreSQL, BigQuery), and which metrics the method enables.

## Workflow

1. **Set up credentials.** Fill in `integrations/<name>/.env` with connection details for your database. Override `credential_env_vars()` in `BaseIntegration` to map credential keys to environment variable names.

2. **Install your database driver.** Add your driver package to `integrations/<name>/requirements.txt` and run `pip install -r integrations/<name>/requirements.txt`.

3. **Start with `BaseIntegration`.** Implement the connection methods first (`credential_env_vars`, `create_connection`, `create_cursor`, `execute_query`, `fetch_all_results`, `close_connection`). These must work before any template test can run.

4. **Validate the connection.** Run `INTEGRATION=<name> pytest -m connection` and confirm all three tests pass before moving on.

5. **Implement templates in sections.** Work through one test file at a time, running tests after each batch:
   - `INTEGRATION=<name> pytest -m metadata` â€” `MetadataQueryTemplates`
   - `INTEGRATION=<name> pytest -m custom_monitors` â€” `CustomSQLMonitorTemplates`
   - `INTEGRATION=<name> pytest -m query_language` â€” `QueryLanguageTemplates` (the bulk of the work, split into `test_ql_prerequisites.py` for prerequisite templates and `test_ql_metrics.py` for metric-specific templates)

6. **Run tests to verify.** Use pytest to check your work:
   ```bash
   # Run all tests
   INTEGRATION=<name> pytest

   # Run a single test file
   INTEGRATION=<name> pytest tests/test_ql_comparison.py

   # Run a single test
   INTEGRATION=<name> pytest tests/test_ql_comparison.py::test_equality -v
   ```

7. **Read failures carefully.** Test failures tell you exactly what went wrong â€” a template rendering error means your Jinja string has the wrong variables, a SQL error means the rendered query is invalid for your database, and an assertion error means the query ran but returned the wrong result.

8. **Review `capabilities.json`.** After each test run, this file is generated at `output/<name>/capabilities.json`. It shows pass/fail status for every template and which metrics your integration supports.
