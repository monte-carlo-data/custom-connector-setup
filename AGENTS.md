# Agent Instructions

## Objective

Implement every method in `integrations/<name>/integration.py` for your target database. Each method either returns a Jinja template string or implements a connection lifecycle hook. The file contains five classes with stub methods — your job is to replace every stub with a working implementation.

## Rules

- **Only edit `integrations/<name>/integration.py`.** Do not modify any files in `tests/`, `conftest.py`, or the plugin. The tests are the spec — make your code pass them, not the other way around.
- **Add your database driver to `integrations/<name>/requirements.txt`** (e.g. `psycopg2-binary`, `snowflake-connector-python`) and install it before implementing `create_connection()`.
- Every template method must return a **Jinja template string**, not raw SQL and not Python logic. For example: `return "AVG({{ field }})"`.
- **Do not edit `capabilities.json`.** This file is auto-generated by the test suite after each run in `output/<name>/capabilities.json` — do not modify it manually.
- Read each method's docstring before implementing it. The docstring documents the Jinja variables the template receives, example implementations for common databases (Snowflake, PostgreSQL, BigQuery), and which metrics the method enables.

## Implementation Phases

Work through phases in order. Each phase has a dedicated prompt file with method details, examples, and test expectations. **Do not advance to the next phase until the current phase's gate tests pass.**

### Phase 1: Connection — `BaseIntegration` (6 methods)

**Prompt:** [`prompts/implement-connection.md`](prompts/implement-connection.md)
**Class:** `BaseIntegration`
**Gate:** `INTEGRATION=<name> pytest -m connection` → 3 tests pass

Implement `credential_env_vars`, `create_connection`, `create_cursor`, `execute_query`, `fetch_all_results`, `close_connection`. Everything else depends on a working connection.

### Phase 2: Metadata — `MetadataQueryTemplates` + `QueryLogCollectionTemplates` (5 methods)

**Prompt:** [`prompts/implement-metadata.md`](prompts/implement-metadata.md)
**Classes:** `MetadataQueryTemplates`, `QueryLogCollectionTemplates`
**Gate:** `INTEGRATION=<name> pytest -m metadata` → core tests pass (up to 9 total, some optional)

Implement `get_databases_query_template`, `get_schemas_query_template`, `get_tables_query_template`, `get_columns_query_template`, `get_query_logs_query_template`.

### Phase 3: Custom Monitors — `CustomSQLMonitorTemplates` (3 methods)

**Prompt:** [`prompts/implement-monitors.md`](prompts/implement-monitors.md)
**Class:** `CustomSQLMonitorTemplates`
**Gate:** `INTEGRATION=<name> pytest -m custom_monitors` → 3 tests pass

Implement `transform_into_count_query_template`, `add_row_limit_template`, `get_count_all_expression_template`.

### Phase 4: Query Language — `QueryLanguageTemplates` (~94 methods)

**Prompt:** [`prompts/implement-ql.md`](prompts/implement-ql.md)
**Class:** `QueryLanguageTemplates`
**Gate:** `INTEGRATION=<name> pytest -m query_language`

The bulk of the work. Start with Core Query Building (CTE, SELECT, FROM, UNION) — most other tests depend on these. See the prompt file for 13 subsections with method tables and test expectations.

## Troubleshooting

When tests fail, consult [`prompts/test-and-fix.md`](prompts/test-and-fix.md) for:
- How to run tests (by phase, by file, single test)
- Interpreting failures (template error, SQL error, assertion error)
- Common patterns (dialect-specific syntax, composed templates)
- Understanding `capabilities.json`

## Workflow Summary

1. Read the phase prompt
2. Implement the methods
3. Run the gate test
4. If tests fail → read the error, consult `prompts/test-and-fix.md`, fix, re-run
5. When the gate passes → move to the next phase
6. After all phases pass, run `INTEGRATION=<name> pytest` and review `output/<name>/capabilities.json`
